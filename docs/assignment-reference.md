# Assignment reference mapping

This document maps each requirement from the assignment to the corresponding files or implementation details in this repository.  It serves as a traceability matrix ensuring that every requested item is accounted for.

| Requirement                                                                                     | Location and implementation details                                                                                                                           |
|-------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Three application containers** – Apache frontend, FastAPI API and PostgreSQL database          | Dockerfiles at `frontend/Dockerfile` and `api/Dockerfile`; Kubernetes manifests under `k8s/` (Deployments, StatefulSet); `docker-compose.yml` for local testing                                    |
| **Use provided gist JavaScript**                                                                | `frontend/src/index.html` is based on the provided gist and extended to fetch `/api/name` and `/api/container-id` and poll `version.txt` for layout refresh                                        |
| **Automatic layout refresh**                                                                    | `frontend/static/version.txt` holds a version string; JavaScript in `frontend/src/index.html` polls this file and reloads the page when it changes                                                |
| **API endpoints** (`/api/name`, `/api/container-id`, `/healthz`, `/readyz`, `/metrics`)           | Implemented in `api/app/main.py` using FastAPI and prometheus‑client                                                                                            |
| **Retrieve container ID**                                                                       | `/api/container-id` parses `/proc/self/cgroup` and returns the container ID along with pod and host names                                                     |
| **Database seeded with default name**                                                           | Alembic migration `api/migrations/versions/001_create_person_table.py` creates the `person` table and inserts the default name (`Dieter Beckers`)                                                    |
| **Update name and reflect change on page**                                                      | `scripts/update_name.sh` modifies the `person` table; page reload triggers a new fetch of `/api/name`                                                        |
| **Health checks and automatic restart**                                                         | Readiness and liveness probes defined in `k8s/api/deployment.yaml` and `k8s/frontend/deployment.yaml`; Kubernetes restarts pods when probes fail                                                    |
| **Multiple API replicas distributed across nodes**                                               | Deployment `k8s/api/deployment.yaml` requests three replicas and defines `topologySpreadConstraints` to spread them across `kubernetes.io/hostname`                                                  |
| **HTTPS with cert‑manager**                                                                     | ClusterIssuers defined in `k8s/cert-manager/clusterissuer.yaml`; ingress TLS configuration in `k8s/ingress/ingress.yaml`                                      |
| **Additional worker node and load‑balanced API**                                                 | Achieved by creating a kubeadm cluster with two worker nodes and deploying the API with multiple replicas; load balancing occurs via Kubernetes Service                                             |
| **Prometheus monitoring**                                                                       | API exposes `/metrics`; ServiceMonitor defined in `k8s/monitoring/service-monitor.yaml`; instructions for installing kube‑prometheus‑stack                                                         |
| **Kubeadm cluster with 1 control plane and 2 workers**                                           | Setup instructions included in `README.md` under “Kubeadm cluster setup”                                                                                      |
| **Helm to install ArgoCD**                                                                      | Values file `helm/argocd-values.yaml` customises the official ArgoCD Helm chart; installation commands provided in README                                     |
| **GitOps with ArgoCD**                                                                          | ArgoCD Application manifest `k8s/argocd/application.yaml` points to this repository; ArgoCD watches and applies changes automatically                        |
| **Scripts for automation**                                                                      | `build.sh`, `push.sh`, `deploy-local.sh`, `deploy-k8s.sh`, `test-local.sh`, `test-k8s.sh`, `package.sh` automate building, deploying and testing                                                  |
| **Local testing instructions**                                                                  | Detailed in `LOCAL_TESTING.md`; includes curl commands and update instructions                                                                                |
| **PDF documentation**                                                                           | Generated file `docs/submission.pdf` summarises the solution and explicitly references every required component                                                |
| **Security best practices**                                                                     | Non‑root containers, minimal base images, secrets via Kubernetes secrets, resource limits and liveness/readiness probes are described in `README.md` and implemented in manifests                  |
| **Packaging into ZIP**                                                                           | `package.sh` creates `db-k8s-stack.zip` containing the entire repository                                                                                    |

The mapping above demonstrates that every assignment requirement has been implemented and can be traced to specific files or instructions in this repository.
